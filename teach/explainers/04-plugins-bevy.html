Les plugins Bevy

Ce qu'est un plugin, ce qu'il « expose » à ton jeu, et un tour des plugins fournis d'office par DefaultPlugins — dont tu utilises déjà la moitié sans le savoir.

1 · C'est quoi 2 · Ce qu'il expose 3 · PluginGroup 4 · Les plugins clés 5 · Configurer

Tu viens d'écrire PlayerPlugin. Un plugin n'est pas une nouvelle notion ECS : c'est simplement du code de configuration de l'App emballé sous un nom, qu'on branche avec add_plugins(...). Toute la puissance de Bevy — fenêtre, rendu, input, temps — arrive par ce seul mécanisme.

01Un plugin = de la config d'App emballée

Le trait Plugin demande une seule méthode, build(&self, app). Dedans, tu appelles les mêmes méthodes que dans main(). Ton PlayerPlugin :

pub struct PlayerPlugin;

impl Plugin for PlayerPlugin {
    fn build(&self, app: &mut App) {
        app.add_systems(Startup, spawn_player)
           .add_systems(Update, move_player);
    }
}

Puis .add_plugins(PlayerPlugin). C'est tout. « Ajouter un plugin » = « exécuter sa méthode build sur l'App ». DefaultPlugins fait pareil, en gros, mais pour ~40 plugins d'un coup.

02Ce qu'un plugin « expose »

Un plugin ne renvoie rien : il enregistre des choses dans l'App, qui deviennent ensuite disponibles pour tous tes systèmes. Ce qu'il peut brancher :

add_systems
des systèmes, dans un schedule (Startup/Update…)
insert_resource
init_resource
des ressources (données globales) → lisibles via Res<T>
add_event
des types d'événements (communication entre systèmes)
register_type
des types pour la réflexion / l'inspecteur
add_plugins
d'autres plugins (composition en cascade)
configure_sets
l'ordre d'exécution des systèmes
Le déclic Quand tu écris keyboard: Res<ButtonInput<KeyCode>> dans un système, cette ressource existe parce que InputPlugin (dans DefaultPlugins) l'a insertée et la met à jour chaque frame. Pareil pour Res<Time> ↔ TimePlugin. Un plugin, c'est le fournisseur ; tes systèmes, les consommateurs.

03PluginGroup : un paquet de plugins

DefaultPlugins n'est pas un plugin mais un groupe (PluginGroup). Il existe aussi MinimalPlugins (le strict minimum : temps + boucle, sans fenêtre ni rendu — utile pour un serveur ou des tests). Sur un groupe, tu peux :

DefaultPlugins
    .set(WindowPlugin { /* config perso */ ..default() })  // reconfigurer un plugin
    .disable::<LogPlugin>()                              // en retirer un
    .add(MyPlugin)                                      // en ajouter un

05Configurer un plugin fourni

Cas le plus courant : régler la fenêtre (titre, taille). On reconfigure WindowPlugin dans le groupe avec .set(...) :

Essaie ça Donne un titre et une taille à la fenêtre de BAGARR. Dans main(), remplace .add_plugins(DefaultPlugins) par : 
.add_plugins(DefaultPlugins.set(WindowPlugin {
    primary_window: Some(Window {
        title: String::from("BAGARR"),
        resolution: (1280.0, 720.0).into(),
        ..default()
    }),
    ..default()
}))
Relance cargo r : la barre de titre affiche « BAGARR » et la fenêtre a la taille voulue. ..default() = « le reste des champs, valeurs par défaut » (on y reviendra, c'est un idiome Rust très courant).

✦Récap

Un plugin = du code de setup d'App emballé (méthode build) qui enregistre systèmes / ressources / events / types. Tes systèmes consomment ensuite ce que les plugins ont fourni (d'où Res<Time>, Res<ButtonInput>…). DefaultPlugins est un groupe de ~40 plugins ; on le configure avec .set(), on en retire avec .disable(). Toi, tu ranges ton jeu en plugins par feature (PlayerPlugin, bientôt EnemyPlugin).
