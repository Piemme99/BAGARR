Ce qui se passe quand tu tapes
cargo run

La compilation Rust, les profils & opt-level, et pourquoi Bevy est long à compiler — pour rendre ton itération rapide en comprenant chaque choix.

1 · Le pipeline 2 · Pourquoi c'est long 3 · opt-level 4 · L'astuce dev/deps 5 · mold & dynamic_linking

Rust est un langage compilé en avance (AOT, ahead-of-time) : tout ton code — et tout Bevy — est traduit en code machine natif avant que le jeu ne se lance. Pas de machine virtuelle ni d'interpréteur au runtime, contrairement à Python ou JavaScript. C'est ce qui rend les jeux Rust rapides… et c'est aussi pourquoi il faut attendre à chaque build.

01Le pipeline de compilation

Entre ton fichier .rs et l'exécutable, deux outils se relaient : rustc (avec LLVM en backend) fait la traduction et l'optimisation ; le linker recolle les morceaux en un binaire. Clique chaque étape pour la déplier.

1
rustc — front-end
Lexing & Parsing
Ton texte source devient un arbre syntaxique (AST). Les erreurs de syntaxe sortent ici.

2
rustc — analyse
Résolution, typage & borrow-check → HIR
Les noms sont résolus, les types vérifiés, et le fameux borrow-checker valide l'ownership et les durées de vie. Aucune donnée du langage n'existe encore en machine : on raisonne sur du sens.
▾
3
rustc — milieu
MIR & monomorphisation
Le code passe en MIR (représentation intermédiaire simplifiée), puis les génériques sont « dépliés » : une copie de code par type concret utilisé. C'est le gros de la charge de travail (voir §2).
▾
4
rustc — backend (LLVM)
Optimisations & code objet
LLVM optimise (inlining, etc. — piloté par opt-level, §3) et émet des fichiers objets .o, un par « codegen unit ». C'est l'étape la plus coûteuse en CPU sur un gros projet.
▾
5
linker (mold / lld / ld)
Édition de liens
Le linker assemble tous les .o + les bibliothèques en un seul exécutable. Refait à chaque build, même pour une ligne changée → cible n°1 de mold (§5).

Repère Tout ce travail atterrit dans le dossier target/. C'est pour ça qu'il grossit énormément (souvent plusieurs Go avec Bevy) : il cache tout le code intermédiaire et final de tes dépendances.

02Pourquoi Rust — et Bevy — est long à compiler

La monomorphisation

Quand tu écris une fonction générique fn f<T>(x: T), Rust ne produit pas une version unique qui marcherait pour tous les T. Il « stamp » une copie compilée distincte pour chaque type concret réellement utilisé : f::<u32>, f::<Enemy>, f::<Vec3>…

C'est un choix de design : ça rend Rust rapide à l'exécution (chaque version est spécialisée, inlinable, zéro coût d'abstraction), au prix du temps de compilation et de la taille du binaire — il y a beaucoup plus de code à générer et optimiser.

Sources : rustc dev guide — Monomorphization · matklad — Fast Rust Builds

Pourquoi Bevy amplifie tout ça

L'ECS de Bevy repose sur des génériques et des traits partout : chaque système, chaque Query<…>, chaque composant déclenche de la monomorphisation. Un petit jeu peut générer des dizaines de milliers de fonctions monomorphisées. Ajoute un arbre de dépendances énorme (rendu wgpu, audio, fenêtrage, maths…) : des centaines de crates. D'où un premier build en minutes — mais une seule fois, ensuite tout est mis en cache.

Modèle mental Le temps de compil vit dans deux mondes. Premier build = compiler Bevy + toutes les deps (lent, une fois). Builds d'itération = recompiler seulement ton code + relinker. Toutes nos optimisations de setup visent surtout le second.

Pourquoi la monomorphisation, qui coûte cher à la compilation, est-elle quand même au cœur de Rust ?

03opt-level : le curseur compile ⟷ run

opt-level décide combien LLVM travaille à l'étape 4. C'est exactement le compromis « long à compiler » contre « lent à tourner ». Bouge le curseur :

0 · aucune optim123 · maxs/z · taille
Temps de compilation
Vitesse au runtime

Que fait concrètement une « optimisation » ? Le levier principal est l'inlining : remplacer un appel de fonction par le corps de la fonction (supprime le coût d'appel et ouvre d'autres optims). S'ajoutent la suppression de code mort, la propagation de constantes, le déroulage de boucles, la vectorisation SIMD… Chaque passe demande à LLVM d'analyser plus longtemps.

Deux axes séparés Compiler = travail fait une fois, avant de jouer. Tourner = travail fait 60 fois par seconde pendant que tu joues. Un binaire opt-level=0 naît vite mais chaque frame coûte cher ; un opt-level=3 met plus longtemps à naître mais chaque frame est bon marché.

Sources : Cargo Book — Profiles · Rust Book §14.1

Les profils dev et release

Un profil = un jeu de réglages. On peut surcharger n'importe quel réglage par profil dans Cargo.toml — ce qu'on va faire à l'étape 4.

Essaie ça Mesure le compromis toi-même. Une fois le projet buildé, compare le lancement des deux profils : 
cargo run              # profil dev : build rapide, jeu potentiellement mou
cargo run --release    # profil release : build lent, jeu fluide
Sur un jeu vide la différence est invisible ; garde ce réflexe pour quand il y aura 500 ennemis à l'écran.

04L'astuce : code léger, dépendances optimisées

C'est la config qu'on a mise dans Cargo.toml :

[profile.dev]
opt-level = 1                 # NOTRE code : un peu optimisé → recompile vite

[profile.dev.package."*"]
opt-level = 3                 # TOUTES les deps (Bevy) : optim max, compilées 1 fois

Le raisonnement, en une image :

  • Ton code change tout le temps → recompilé à chaque build. On le garde à opt-level = 1 : assez optimisé pour ne pas ramer, assez léger pour recompiler vite.
  • Bevy & les deps ne changent jamais (versions figées) → compilés une seule fois puis cachés. Autant les pousser à opt-level = 3 : ça n'allonge que le premier build, et ensuite le moteur tourne à pleine vitesse même en dev.

Le "*" dans [profile.dev.package."*"] signifie « s'applique à tous les packages » (donc toutes les dépendances). On cible ainsi wgpu & cie, les vrais gouffres de perf, pas seulement bevy.

Sources : Cargo Book — Profiles (overrides) · Bevy — Setup

05mold & dynamic_linking : écraser le coût du linking

mold — un linker plus rapide

Rappel de l'étape 5 : à chaque build d'itération, le linker recolle tous les .o et les grosses libs de Bevy. Avec le linker GNU par défaut (ld), c'est lent sur un projet de la taille de Bevy. mold est un linker moderne, massivement parallèle, souvent plusieurs fois plus rapide. On le branche dans .cargo/config.toml :

[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=mold"]

Ici clang (déjà installé chez toi) ne sert que de « chef d'orchestre » qui invoque mold. rustc parle à clang, clang appelle mold pour lier.

bevy/dynamic_linking — ne plus ré-embarquer Bevy

Par défaut Bevy est lié statiquement : tout son code est copié dans ton exécutable à chaque build (énorme à relinker). La feature dynamic_linking compile Bevy en bibliothèque partagée (.so), chargée au lancement. Ton exécutable devient minuscule et le linker n'a plus à ré-embarquer tout Bevy → itération encore plus rapide.

⚠ Ne ship pas comme ça Bevy déconseille de distribuer avec dynamic_linking : il faut livrer libbevy_dylib à côté, ça empêche certaines optimisations et peut grossir le jeu. On le veut donc uniquement en développement.

Plutôt que de le figer dans Cargo.toml, on l'active via un alias Cargo (aussi dans .cargo/config.toml) :

[alias]
r = "run --features bevy/dynamic_linking"

→ tu tapes cargo r pour développer (rapide, dynamique), et cargo build --release reste propre et statique pour une vraie version. On choisit le compromis par commande, pas globalement.

Sources : Bevy — Setup / Enable Fast Compiles · mold (rui314) · Bevy Discussion #8991

Essaie ça Vois où part le temps de compil. Cargo génère un rapport HTML détaillé du build : 
cargo build --timings
Il t'affiche le chemin d'un fichier target/cargo-timings/cargo-timing.html : ouvre-le, tu verras chaque crate et le temps passé à la compiler. Regarde comme wgpu & bevy dominent.
Pourquoi active-t-on dynamic_linking via un alias cargo r plutôt que dans Cargo.toml ?

✦Récap en une phrase

Compiler, c'est traduire et optimiser ton code + tout Bevy en machine natif ; c'est long surtout à cause de la monomorphisation des génériques de Bevy et de la taille de l'arbre de deps ; opt-level est le curseur entre « vite compilé / lent à tourner » et l'inverse ; on optimise les deps à fond une seule fois tout en gardant notre code léger, et mold + dynamic_linking écrasent le coût de l'étape de linking à chaque itération.
