Le modèle ECS de Bevy

Entity, Component, System, World, App, Schedule, Commands, spawn, Bundle, required components… tout le vocabulaire de ton main.rs, en une carte mentale.

1 · La grande idée 2 · Entity · Component · World 3 · System 4 · App · Plugin · Schedule 5 · Commands · spawn 6 · Bundle · required

Tout Bevy repose sur un seul patron d'architecture : ECS — Entity, Component, System. Une fois ces trois mots en place, tout le reste (App, Commands, spawn…) n'en est qu'un outillage. C'est le concept à internaliser.

01La grande idée : composition de données, pas héritage

Dans une approche objet classique, tu ferais peut-être class Enemy extends Character extends GameObject. Bevy prend le chemin inverse : un objet de jeu n'est pas une classe, c'est un sac de composants (des données) assemblés, et la logique vit à l'extérieur, dans des fonctions qui balaient tous les objets partageant certaines données.

Entity — un simple identifiant (un numéro). Ne contient aucune donnée par lui-même. C'est « la chose », le joueur, un ennemi, une balle.
Component — une donnée attachée à une entité : position, vie, vitesse, couleur… En Bevy, un simple struct avec #[derive(Component)].
System — une fonction de logique qui s'exécute sur toutes les entités portant certains composants (ex : « déplace tout ce qui a Position + Velocity »).

Sources : Bevy — Quick Start : ECS · Cheat Book — Entities & Components

02Entity + Component + World = une base de données

Le meilleur modèle mental : une table. Chaque entité est une ligne (son id), chaque type de composant est une colonne. Une case cochée = cette entité possède ce composant. Le World est la base entière (toutes les entités et leurs composants).

👆 Clique les cases pour donner/retirer un composant. Un « ennemi » = une entité avec Position + Velocity + Health + Enemy. Un système ne verra que les lignes ayant les colonnes qu'il demande.
Pourquoi ce design Ranger les données par composant (par colonne) rend le parcours ultra-rapide pour le CPU (mémoire contiguë) et permet à Bevy de faire tourner des systèmes en parallèle automatiquement quand ils ne touchent pas aux mêmes données. C'est la raison de perf derrière l'ergonomie.

03System : de la logique injectée

Un système est juste une fonction. Sa signature est un contrat : tu déclares ce dont tu as besoin en paramètres, et Bevy te le fournit automatiquement (injection de dépendances). Ton setup en est un :

fn setup(mut commands: Commands) { ... }
//              ↑ "j'ai besoin de pouvoir créer des entités" → Bevy fournit Commands

Les paramètres possibles (on les découvrira au fil des phases) : Commands (créer/ détruire), Query<…> (lire/écrire les composants d'entités existantes — la prochaine brique, en Phase 1), Res<T> (lire une ressource globale, ex. le temps), etc.

04App, Plugin & Schedule : qui orchestre, et quand

App — l'application entière. On la construit (builder) en y branchant des plugins et des systèmes, puis .run() lance la boucle de jeu.
Plugin — un paquet de systèmes + config prêt à brancher. DefaultPlugins t'apporte fenêtre, rendu, entrées… Plus tard tu écriras tes propres plugins (PlayerPlugin, EnemyPlugin) pour ranger ton code.
Schedule — quand un système tourne. Les deux qu'on utilisera : Startup (une fois au lancement) et Update (à chaque frame).

05Commands, spawn & despawn

Commands est une file d'instructions pour les changements structurels : créer une entité (spawn), la détruire (despawn), lui ajouter/retirer un composant. Tu l'as utilisé : commands.spawn(Camera2d).

Pourquoi « file » et pas immédiat Modifier la structure du World en plein milieu d'un système (pendant que d'autres le lisent, potentiellement en parallèle) serait dangereux. Bevy met donc tes commandes en attente et les applique à un point de synchronisation sûr, à la fin. D'où le nom : ce sont des commandes à exécuter, pas des actions immédiates.

Source : Cheat Book — Commands

06Bundle & Required Components

Un Bundle = une collection de composants qu'on ajoute d'un coup à une entité. Un tuple de composants est un bundle :

commands.spawn((Player, Transform::default(), Sprite::from_image(...)));
//              └──────────────── un tuple = un Bundle : 3 composants d'un coup ────┘

Le twist Bevy 0.15+ : les required components

Avant, on utilisait des bundle-structs tout faits (Camera2dBundle, SpriteBundle…). Ils sont dépréciés depuis Bevy 0.15, remplacés par les required components : un composant peut déclarer, via #[require(...)], d'autres composants qui seront ajoutés automatiquement si tu ne les fournis pas. C'est exactement pourquoi spawn(Camera2d) tout seul t'a donné une caméra fonctionnelle.

commands.spawn(Camera2d);
■ ce que tu écris · ■ ajouté automatiquement via #[require] (vérifié dans le source de Bevy 0.19)

Sources : Bevy 0.15 — Required Components · Cheat Book — Bundles · source bevy_camera-0.19.0

En Bevy 0.19, quelle est la façon idiomatique de faire apparaître une caméra 2D ?
Essaie ça Touche à une ressource globale. Le fond gris, c'est une Resource nommée ClearColor (une donnée unique, pas attachée à une entité). Ajoute cette ligne dans le builder de ton App, avant .run() : 
.insert_resource(ClearColor(Color::srgb(0.1, 0.2, 0.35)))
Relance avec cargo r : le fond devient bleu nuit. Tu viens de manipuler le 4ᵉ grand concept ECS — la Resource (donnée globale unique), en complément des composants (données par entité).

✦La carte mentale en une phrase

Une Entity est un id ; on lui accroche des Components (données) ; le World est la base qui contient tout ça ; les Systems (fonctions) lisent/écrivent ces composants ; l'App assemble des Plugins et range les systèmes dans des Schedules (Startup/Update) ; Commands crée/détruit les entités (spawn/despawn) de façon différée ; un Bundle (souvent un tuple) ajoute plusieurs composants d'un coup, et les required components (#[require]) en tirent d'autres automatiquement.
