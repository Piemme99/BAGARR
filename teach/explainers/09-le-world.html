Le World — la base de données de ton jeu

Tout l'état du jeu vit dans un seul conteneur que tu n'as jamais vu. Comprendre le World, c'est comprendre d'un coup les signatures de systèmes, Commands, le parallélisme… et FromWorld.

1 · La base de données 2 · Les vues prêtées 3 · Commands, la file 4 · La clé maîtresse 5 · Les 3 niveaux

Depuis le début tu écris des systèmes qui « reçoivent » des Query, des Res, des Commands… mais reçoivent depuis où ? La réponse est toujours la même : depuis le World, le conteneur unique qui possède tout l'état de BAGARR. Cette fiche le rend visible.

01Le World est une base de données en mémoire

Le World possède tout :

  • toutes les entités — ton joueur, chaque ennemi, le texte du HUD, la caméra ;
  • tous les composants attachés à ces entités — Transform, Sprite, Enemy… ;
  • toutes les resources — les singletons globaux : Kills, Time, Assets<Mesh>, ton futur EnemyAssets.

L'analogie qui tient la route : une base de données relationnelle en mémoire. Les entités sont des lignes (juste un identifiant), les composants des colonnes, les resources des tables à une seule ligne. Et les systèmes ? Des requêtes qui déclarent à l'avance ce qu'elles lisent et ce qu'elles écrivent.

Quand tu écris App::new(), Bevy crée un World à l'intérieur de l'App. Il existe depuis la première ligne de ton main.rs — tu ne l'as jamais vu parce que tu n'y accèdes qu'indirectement. Toute la suite de la fiche décrit les trois façons d'y accéder.

02Les systèmes ne touchent jamais le World — ils déclarent des vues

Reprends un de tes systèmes (hud.rs) :

fn update_hud(kills: Res<Kills>, mut query: Query<&mut Text, With<KillsText>>)

Ces paramètres ne sont pas des données que tu passes : ce sont des déclarations d'accès (fiche 05). Avant d'exécuter le système, l'ordonnanceur lit la signature et lui prête une vue étroite sur le World : exactement ce qu'il a demandé, rien de plus.

Pourquoi cette indirection ? Le parallélisme. Puisque Bevy connaît les accès de chaque système, il peut en faire tourner plusieurs en même temps dès qu'ils ne se marchent pas dessus. C'est la règle d'emprunt de Rust, appliquée par l'ordonnanceur aux données du World :

N lecteurs en même temps  OU  1 seul écrivain

— jamais un écrivain avec qui que ce soit d'autre sur la même donnée.

Exemple concret dans BAGARR : update_hud (lit Kills, écrit les Text) et move_player (écrit le Transform du joueur) touchent des données disjointes → Bevy a le droit de les exécuter simultanément sur deux threads. Tu n'as rien demandé : la signature suffit.

Et c'est aussi pour ça que ton Without<Player> dans update_enemies est obligatoire : sans lui, une query &mut Transform et une query &Transform pourraient viser la même entité — un écrivain + un lecteur sur la même donnée, interdit. Le filtre prouve à Bevy que les deux ensembles d'entités sont disjoints.

Deux systèmes prennent chacun Query<&Transform, With<Enemy>> (lecture seule, mêmes entités). Bevy peut-il les paralléliser ?

03Commands : la file d'attente des changements de structure

Une vue prêtée permet de lire et modifier des valeurs existantes. Mais il y a une chose qu'elle ne permet pas : changer la structure du World — spawner une entité, en despawner une, insérer une resource. Pourquoi ? Parce que pendant que ton système tourne, d'autres systèmes tournent en parallèle avec leurs propres vues ; modifier la structure sous leurs pieds les invaliderait.

D'où Commands : ce n'est pas un accès au World, c'est une file d'attente. Quand tu écris commands.spawn((Enemy, ...)) dans spawn_enemies, rien n'est créé sur le champ. Bevy note « à faire », et applique toute la file un peu plus tard, à un point de synchronisation — un moment où il reprend l'accès complet et exclusif au World entre deux groupes de systèmes.

Conséquence concrète Une entité spawnée via Commands n'est pas visible par les queries des systèmes qui tournent avant le prochain point de synchro. Si un jour un ennemi « n'existe pas encore » dans un autre système de la même frame, tu sauras pourquoi — ce n'est pas un bug, c'est le différé.

04&mut World : la clé maîtresse

Dernier niveau : recevoir &mut World directement. C'est l'accès exclusif à absolument tout — sans vue, sans file d'attente… et donc sans parallélisme : rien d'autre ne tourne pendant ce temps.

Tu le croises à deux endroits :

  • FromWorld — le trait derrière init_resource. C'est « Default, mais avec le droit de fouiller le World pendant la construction ». Indispensable quand une resource ne peut pas se construire à partir de rien — ton EnemyAssets doit aller piocher dans Assets<Mesh> : 
    impl FromWorld for EnemyAssets {
    fn from_world(world: &mut World) -> Self {
        // seul au monde : accès direct, séquentiel, à n'importe quelle resource
        let mut meshes = world.resource_mut::<Assets<Mesh>>();
        // ...
    }
}
  • Les systèmes exclusifs — un système dont le paramètre est &mut World. Rare, réservé aux cas où on doit tout manipuler d'un coup (sauvegarde, reset complet…).

Le prix de la clé maîtresse : tout est à l'arrêt. Parfait pour de l'initialisation qui tourne une fois au démarrage ; mauvaise idée pour du code de frame.

Borrow checker Dans from_world, tu ne peux pas tenir deux world.resource_mut::<…>() en même temps : deux emprunts mutables du même world, E0499. La parade est simple — termine l'un (récupère ton Handle, qui lui n'emprunte rien), puis ouvre l'autre. Séquentiel, pas simultané.

05Les trois niveaux d'accès, côte à côte

Res / Query

Vue étroite prêtée

Parallélisable. Le quotidien : 99% de tes systèmes.

Commands

File différée

Changements de structure (spawn, despawn, insert_resource) appliqués au prochain point de synchro.

&mut World

Clé maîtresse

Exclusif, tout à l'arrêt. FromWorld, systèmes exclusifs. Init, pas du code de frame.

Pourquoi ne pas écrire toute la logique du jeu dans des systèmes exclusifs (&mut World) ? Ce serait plus simple, non ?
Essaie ça Provoque le conflit toi-même. Dans update_enemies (enemy.rs), supprime le Without<Player> de la query des ennemis, puis lance cargo r. Tu obtiens un panic au démarrage mentionnant l'erreur B0001 : Bevy refuse un système dont les deux queries pourraient viser la même entité avec un accès &mut + &. Lis le message en entier — il t'explique lui-même les deux parades (Without, ou ParamSet). Puis remets le filtre. Tu viens de voir l'ordonnanceur défendre la règle « N lecteurs OU 1 écrivain » à l'exécution.

✦Récap

Le World est la base de données unique qui possède tout l'état du jeu. Les systèmes n'y touchent jamais directement : leurs signatures déclarent des accès, et l'ordonnanceur leur prête des vues étroites — ce qui lui permet de paralléliser tout ce qui ne se conflit pas (N lecteurs OU 1 écrivain). Les changements de structure passent par Commands, une file différée appliquée aux points de synchronisation. Et &mut World est la clé maîtresse — exclusive, tout à l'arrêt — qu'on ne sort que pour l'initialisation (FromWorld) ou les rares systèmes exclusifs.
