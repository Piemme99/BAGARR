Communication & découplage entre systèmes

Les systèmes ne s'appellent pas entre eux — ils parlent à travers le World. Comment coordonner à l'échelle sans emmêler ton code.

1 · Ils ne s'appellent pas 2 · Les 3 canaux 3 · Messages 4 · L'ordre 5 · Observers 6 · Quoi utiliser

01Le recadrage : les systèmes ne s'appellent pas

En Bevy, tu n'as pas de graphe « le système A appelle B appelle C ». Les systèmes sont indépendants : Bevy les ordonnance, ils ne se référencent jamais. Ils communiquent par le World.

Nuance importante Ton attack qui appelle kill_enemies_in_circle, ce n'est pas du couplage entre systèmes : kill_enemies_in_circle est une fonction helper (pas un système), appelée dans un système. Décomposer un système en fonctions pures = sain. Le « web de systèmes qui s'appellent » que tu crains n'existe pas par construction.

02Les canaux de communication

Un système produit de la donnée, un autre la consomme, sans se connaître :

CanalPour quoiCouplage
Componentétat attaché à une entité (« qui a quoi »)très faible — on agit sur « toute entité avec X »
Resourceétat global partagé (score, config, timers)faible
Message« un événement est arrivé » → 0..N réactions (bufferisé)le plus faible — producteur ↔ consommateurs anonymes
Observer (Event)réaction immédiate & ciblée à un événementfaible, réactif

03Messages : « X est arrivé → plusieurs réactions »

Le levier anti-couplage central. Bascule la démo :

API Bevy 0.19 (vérifiée dans le source) — un producteur écrit, des consommateurs indépendants lisent :

#[derive(Message)]                       // ⚠️ Message, PAS Event (renommé en 0.19)
struct EnemyKilled { position: Vec2 }

// enregistrement : app.add_message::<EnemyKilled>();

// PRODUCTEUR — annonce, sans rien savoir des réactions :
fn attack(mut killed: MessageWriter<EnemyKilled>, ...) {
    killed.write(EnemyKilled { position });
}

// CONSOMMATEURS — chacun réagit, aucun ne connaît les autres :
fn update_score(mut r: MessageReader<EnemyKilled>, mut kills: ResMut<Kills>) {
    for _ in r.read() { kills.0 += 1; }
}
// fn play_death_sound(...)   fn spawn_particles(...)  ← ajout = nouveau système, rien à toucher ailleurs
Vocabulaire 0.19 C'est Message / MessageWriter / MessageReader / .write() / .read() / add_message. Les tutos (et le Cheat Book) disent encore Event / EventWriter / send → périmé, même concept, autres noms.
Piège du décalage d'1 frame Un message écrit est lu au prochain passage du lecteur. Si le lecteur tourne avant l'écrivain dans la frame, il lira le message la frame suivante (1 frame de retard). Si tu as besoin du même-frame, force l'ordre (§4).

Source : Cheat Book — Events (concept ; noms à traduire) · source vérifiée bevy_ecs/src/message/*

04L'ordre : dépendance déclarative, pas un appel

Si B a besoin de ce que A produit dans la même frame, tu ne fais pas « B appelle A » : tu déclares l'ordre.

.add_systems(Update, (a, b).chain())        // a, puis b
.add_systems(Update, b.after(a))          // b après a
.add_systems(Update, a.before(b))         // équivalent

+ les SystemSet pour grouper et ordonner à plus grande échelle. Bevy fait tourner en parallèle tout ce qui n'a pas de conflit d'accès (fiche 05) ; l'ordre n'est imposé que là où tu le demandes.

Source : Cheat Book — System Order

05Observers : la réaction immédiate (le nouveau sens d'Event)

Là où un Message est bufferisé (lu à la prochaine exécution du lecteur), un observer réagit immédiatement quand un Event est déclenché, souvent ciblé sur une entité. API 0.19 (vérifiée) :

#[derive(Event)]
struct Damage(u32);

// un observer = une fonction qui prend On<E> ; on l'attache avec app.add_observer(...)
fn on_damage(trigger: On<Damage>, ...) { /* réagit tout de suite */ }

Résumé : Message = « diffuse à qui veut, traité au prochain tour » (score, sons, stats…) ; Observer/Event = « réagis maintenant, précisément » (ex. appliquer des dégâts à cette entité). On approfondira l'API si on s'en sert.

Source vérifiée : bevy_ecs/src/observer/* (On<E>, add_observer)

06Quoi utiliser, quand

BesoinOutil
Un état durable attaché à des entitésun Component (+ un système qui le lit)
Un état global (score, réglages)une Resource
« Un truc est arrivé », plusieurs réactions possiblesun Message (le cas « mort → score + son + particules »)
Réagir tout de suite, sur une entité préciseun Observer (Event + add_observer)
« B doit voir ce que A a fait cette frame »ordre : .after/.chain
Découper des logiques d'une même famillefonctions helpers dans un système (données pures)
À la mort d'un ennemi tu veux : +1 au score, un son, des particules, et plus tard un drop d'XP. Comment garder ça découplé ?

✦Récap

Les systèmes ne s'appellent pas : ils produisent/consomment de la donnée via Components, Resources, Messages (bufferisés, 0..N lecteurs) et Observers (réaction immédiate). Pour « un événement → plusieurs réactions » → un Message (ajouter une réaction = ajouter un système, sans toucher au producteur). Pour « B après A » → l'ordre déclaratif (.after/.chain). Pas de graphe d'appels à suivre : c'est ce qui fait que l'ECS scale. Pense « flux de données », pas « appel ».
